Why Use LogScrub?

Every day, developers and IT professionals need to share logs for debugging, support tickets, bug reports, and collaboration. But logs often contain sensitive information that shouldn't be shared.

Common Scenarios

Sharing Logs with AI Assistants

AI tools like ChatGPT, Claude, and Copilot are incredibly useful for debugging and analyzing logs. But pasting raw logs means sending customer emails, IP addresses, API keys, and other sensitive data to third-party services.

Solution: Scrub your logs first. The AI can still understand error patterns, stack traces, and timing issues without seeing real customer data. A log showing [EMAIL-1] failed to authenticate from [IPV4-1] is just as useful for debugging as one with real values.

Filing Bug Reports & Support Tickets

When reporting issues to software vendors or open-source projects, you often need to include logs. These logs may contain your company's internal hostnames, user data, or credentials that were accidentally logged.

Solution: LogScrub removes PII while preserving the structure and timing that developers need to diagnose issues.

Sharing with Colleagues & Contractors

Not everyone on your team needs to see production customer data. When sharing logs for code review or debugging, sanitizing first follows the principle of least privilege.

Compliance & Data Protection

Regulations like GDPR, HIPAA, and CCPA restrict how personal data can be shared and processed. Sanitizing logs before sharing helps maintain compliance and reduces your data exposure footprint.

Why Client-Side Processing Matters

Many log sanitization tools require uploading your logs to a server. This defeats the purpose — you're sharing sensitive data with yet another third party.

LogScrub processes everything in your browser using WebAssembly. Your logs never leave your device. You can even use it offline or on an air-gapped machine.

Preserving Log Usefulness

The goal isn't just to remove data — it's to keep logs useful for their intended purpose:

  • Consistency Mode ensures the same email always becomes [EMAIL-1], so you can still trace a user's journey through the logs
  • Time Shift lets you anonymize timestamps while preserving the relative timing between events
  • Selective Rules let you keep non-sensitive data like UUIDs or timestamps when they're needed for debugging

Getting Started

LogScrub helps you remove personally identifiable information (PII) and sensitive data from log files, debug output, and text documents before sharing them.

Quick Start

  1. Paste or upload your log content into the "Original" pane
  2. Review the detection rules in the left panel (enabled rules are checked)
  3. Click "Scrub" or press ⌘/Ctrl + Enter
  4. Copy or download the scrubbed output
Before (Original)
2024-01-15 10:23:45 INFO User login successful email: john.doe@example.com ip: 192.168.1.105 session: sid=abc123def456789
After (Scrubbed)
2024-01-15 10:23:45 INFO User login successful email: [EMAIL-1] ip: [IPV4-1] session: [SESSION_ID-1]

How It Works

LogScrub uses pattern matching to identify sensitive data in your text. Each detection rule has a regular expression (regex) that matches specific data formats.

Processing Pipeline

  1. Pattern Matching — Each enabled rule's regex is applied to find matches
  2. Replacement — Matched text is replaced according to your chosen strategy
  3. Consistency — When enabled, identical values get identical replacements
Tip: Use the "Analyze" feature to preview what will be detected before sanitizing. This also suggests disabled rules that would match content in your text.
Important: Only enable rules for data types you expect to find in your content. Enabling all rules will likely cause false positives — for example, the "US Zip Code" pattern matches any 5-digit number, and "Date" patterns may match version numbers. Be selective to get accurate results.

Replacement Strategies

Choose how detected PII should be replaced:

Label

Replaces with a descriptive label and counter.

john@example.com → [EMAIL-1]

Fake

Replaces with realistic fake data.

john@example.com → user42@redacted.net

Redact

Replaces with asterisks matching the original length.

john@example.com → ****************

Template

Custom replacement format using variables.

{TYPE}, {n}, {len}

Consistency Mode

When enabled, the same input value always produces the same replacement. This preserves relationships in your data:

With Consistency Mode
User [EMAIL-1] logged in User [EMAIL-1] viewed page User [EMAIL-2] logged in
Without Consistency Mode
User [EMAIL-1] logged in User [EMAIL-2] viewed page User [EMAIL-3] logged in

Custom Rules

Create your own detection rules using regular expressions for patterns specific to your organization.

Adding a Custom Rule

  1. Click the "+ Regex" button in the rules panel
  2. Enter a descriptive name for the rule
  3. Enter your regex pattern (JavaScript syntax)
  4. Click "Add Rule"

Example: Company Employee IDs

Name: Employee ID

Pattern: EMP-[0-9]{6}

Matches: EMP-123456, EMP-000001

Example: Internal Hostnames

Name: Internal Servers

Pattern: \b[a-z]+-(?:prod|staging|dev)-[0-9]+\.internal\.company\.com\b

Matches: api-prod-01.internal.company.com

Tip: Test your regex pattern using the "View Pattern" button (⚙) which includes a pattern tester.

Plain Text Patterns

For exact text matches that don't require regex (like specific hostnames or identifiers), use Plain Text patterns.

Adding a Plain Text Pattern

  1. Click the "+ Text" button in the rules panel
  2. Enter a label for the pattern
  3. Enter the exact text to match
  4. Click "Add Pattern"

When to Use Plain Text vs Regex

Use Plain Text Use Regex
Specific hostnames: db-master.prod.internal Pattern-based hostnames: db-\d+\.prod\.internal
Known usernames: admin_system Username patterns: user_[a-z]+_[0-9]+
Fixed identifiers: ACME-CORP Variable IDs: ACME-[A-Z]{3}-[0-9]+

Time Shift

Timestamps in logs can reveal when incidents occurred, work patterns, or timezone information. Time Shift lets you anonymize temporal data while preserving the relative timing between events.

Note: Time Shift only appears when LogScrub detects timestamps in your input.

Supported Timestamp Formats

LogScrub automatically detects common timestamp formats:

  • ISO 8601 — 2024-01-15T10:30:00Z, 2024-01-15T10:30:00.123+00:00
  • ISO Date — 2024-01-15
  • Apache error log — [Sat Aug 12 04:05:51 2006]
  • Apache access log — [17/May/2015:10:05:03 +0000]
  • Syslog — Jan 15 10:30:00
  • US Format — 01/15/2024 10:30:00

Offset Mode

Shift all timestamps by a fixed amount. Useful when you want to obscure the actual date/time while keeping the duration between events accurate.

Original
2024-01-15T10:30:00Z Request started 2024-01-15T10:30:05Z Database query 2024-01-15T10:30:07Z Response sent
Shifted by -48 hours
2024-01-13T10:30:00Z Request started 2024-01-13T10:30:05Z Database query 2024-01-13T10:30:07Z Response sent

Start From Mode

Set the first timestamp to a specific date/time. All subsequent timestamps shift by the same amount, preserving relative timing.

This is useful when you want logs to appear to start from a neutral date like 2000-01-01.

Scope: Line Start vs All Timestamps

Logs often contain two types of timestamps:

  • Log timestamps — At the start of each line, indicating when the event occurred
  • Content dates — Within the log message itself (DOBs, appointment dates, expiry dates, etc.)
Example log with both types
2024-01-15T10:30:00Z User created. DOB: 1985-03-22, subscription expires: 2025-06-01 2024-01-15T10:31:00Z Appointment scheduled for 2024-02-20 at 14:00

The Scope setting controls which timestamps are shifted:

Scope Shifts Leaves Alone Best For
Line Start Log timestamps at line beginning Dates within message content Shifting log timing while sanitizing content dates with detection rules
All Timestamps Every timestamp found Nothing When all dates should shift together (e.g., test data generation)
Line Start mode (shifted -30 days)
2023-12-16T10:30:00Z User created. DOB: 1985-03-22, subscription expires: 2025-06-01 2023-12-16T10:31:00Z Appointment scheduled for 2024-02-20 at 14:00
All Timestamps mode (shifted -30 days)
2023-12-16T10:30:00Z User created. DOB: 1985-02-20, subscription expires: 2025-05-02 2023-12-16T10:31:00Z Appointment scheduled for 2024-01-21 at 14:00
Tip: Use "Line Start" scope with date/time detection rules enabled. This way, log timestamps get shifted while content dates like DOBs get scrubbed to [DATE-1].

Interaction with Date/Time Rules

When Time Shift is enabled, date/time detection rules (DateTime ISO, DateTime CLF, Date, Time, etc.) are automatically skipped to prevent conflicts. This ensures timestamps are shifted rather than replaced with labels.

Working with Network Captures

LogScrub can scrub network packet captures (PCAP files), but they must first be converted to text format.

Converting PCAP to Text

Use standard packet analysis tools to export your capture as readable text:

Using tcpdump

tcpdump -r capture.pcap -tttt > capture.txt

The -tttt flag prints full timestamps. Add -v or -vv for more detail.

Using tshark (Wireshark CLI)

tshark -r capture.pcap > capture.txt

For specific protocols or fields:

tshark -r capture.pcap -T fields -e ip.src -e ip.dst -e tcp.port > capture.txt

Using Wireshark

  1. Open the PCAP file in Wireshark
  2. Go to File → Export Packet Dissections → As Plain Text
  3. Save the text file, then upload to LogScrub

What Gets Detected

Network capture text typically contains:

Data Type Detection Rule Example
IP addresses with ports IPv4, IPv6 192.168.1.1:8080, [::1]:443
MAC addresses MAC Address 00:1A:2B:3C:4D:5E
Hostnames in DNS/HTTP Hostname, URL api.example.com
HTTP headers Various token rules Auth tokens, cookies, etc.
Tip: IP addresses with port numbers (like 192.168.1.1:8080 or [2001:db8::1]:443) are automatically matched by the IPv4 and IPv6 rules.

File Size & Performance

LogScrub runs entirely in your browser, so performance depends on your device's capabilities. Here's what to expect:

File Size Performance
< 10 MB Fast, smooth processing
10–50 MB Works well, may take a few seconds
50–100 MB Slower processing, may take 10–30 seconds
> 100 MB May cause browser memory issues on some devices

Technical Limits

  • JavaScript string limit — Browsers typically support strings up to 512MB–1GB
  • Browser tab memory — Usually 1–4GB depending on your device
  • WebAssembly memory — Can use up to 4GB

Tips for Large Files

  • Close other browser tabs to free up memory
  • Use a desktop browser rather than mobile for very large files
  • Consider splitting extremely large log files before processing
  • The virtual scrolling feature (for files with 500+ lines) helps keep the UI responsive

Privacy & Security

Important: LogScrub runs 100% in your browser. Your data is never uploaded to any server.

How We Protect Your Data

  • Client-side processing — All scrubbing happens in your browser using WebAssembly
  • No server communication — Your text never leaves your device
  • No analytics on content — We don't track what you scrub
  • Local storage only — Presets and settings are stored in your browser
Disclaimer: LogScrub does not guarantee detection of all PII. Pattern-based detection has inherent limitations and may miss sensitive data in unexpected formats, custom identifiers, or context-dependent information. Always review your scrubbed output before sharing. You are responsible for ensuring adequate sanitization of your data.

Open Source

LogScrub is open source. You can inspect the code, run it locally, or host your own instance.

Best Practices

  • Always review the scrubbed output before sharing
  • Use "Analyze" first to understand what will be detected
  • Enable additional rules if your logs contain sensitive data types that aren't detected by default
  • Add custom patterns for organization-specific identifiers
  • Consider using "Redact" strategy for highly sensitive data to completely obscure the original values
  • Use the "Changed only" filter to quickly review all lines that were modified
